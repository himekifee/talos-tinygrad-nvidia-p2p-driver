# Talos NVIDIA P2P: Engineering Findings

Reference document for contributors working on custom NVIDIA kernel module extensions for Talos Linux. These findings come from building, debugging, and deploying a P2P-patched NVIDIA open GPU kernel module as a Talos system extension against Talos v1.13.0-beta.1 with the NVIDIA 595.58.03 production driver track.

## The Three-Repo Architecture

Siderolabs splits the Talos ecosystem across three repos with distinct roles:

| Repo | Layout | Role |
|---|---|---|
| [`siderolabs/talos`](https://github.com/siderolabs/talos) | Standard Go project (`cmd/`, `pkg/`, `api/`) | OS runtime and API |
| [`siderolabs/pkgs`](https://github.com/siderolabs/pkgs) | Flat bldr package dirs (`kernel/`, `nonfree/`, ...) | Produces base OCI package images (kernel, drivers, firmware) |
| [`siderolabs/extensions`](https://github.com/siderolabs/extensions) | Domain-category dirs (`drivers/`, `nvidia-gpu/`, ...) | Produces system extension OCI images that overlay onto Talos |

**This repo is an extensions fork.** It follows the extensions layout. Don't create `pkgs/` or `kernels/` directories at root ... that would be a layout category error.

### Two-tier NVIDIA build model

Kernel module compilation is a *pkgs* concern. Extensions just wrap pre-built images.

**Tier 1 (pkgs):** Downloads the NVIDIA redist archive, compiles kernel modules against the kernel build tree (`SYSSRC=/src`), produces a `-pkg` image.

**Tier 2 (extensions):** Pulls the `-pkg` image, copies compiled `.ko` files into `/rootfs/usr/lib/modules/`, produces the system extension image.

The P2P extension breaks this separation by necessity: it needs to *patch* the source before compiling. So the local repo imports minimal kernel build stages from `siderolabs/pkgs` to get a working kernel build tree (`/src`).

## Validated Runtime Fix

The critical Talos-specific fix is forcing donor-equivalent RM registry overrides at module load time through `nvidia.conf`:

```conf
options nvidia NVreg_RegistryDwords="ForceP2P=17;RMForceP2PType=1;RMPcieP2PType=1;PeerMappingOverride=1;RMForceStaticBar1=1"
```

That configuration ships from:

- `nvidia-gpu/nvidia-modules-p2p/pkg/production/files/nvidia.conf`

and is installed into the sysext rootfs at:

- `/rootfs/usr/local/lib/modprobe.d/nvidia.conf`

Why this matters:

- The donor patch must still be compiled into the kernel modules.
- On Talos, that patch alone was not sufficient to make runtime P2P capability checks succeed.
- The working build requires the RM registry overrides to be present at module load time so the driver exposes the expected P2P behavior.

### Validated runtime outcome

With the patched modules and the registry overrides live:

- `nvidia-smi topo -p2p r` reports `OK`
- `cuDeviceCanAccessPeer` reports `1` in both directions
- `cuCtxEnablePeerAccess` returns `CUDA_SUCCESS` in both directions
- `p2pBandwidthLatencyTest` succeeds with working peer connectivity

### `nvidia-smi topo -p2p a` nuance

`nvidia-smi topo -p2p a` is not a reliable pass/fail signal for this setup. A working deployment can still report `NS` for atomics while read capability, CUDA peer access, and the runtime bandwidth/latency test all succeed. Treat `topo -p2p r`, CUDA peer-access APIs, and `p2pBandwidthLatencyTest` as the real verification signals.

## Kernel Module Signing Lineage

**The single most important constraint.** Every `.ko` file must be signed by the same key that compiled the running kernel. Talos enforces this at module load time.

What this means in practice:

1. The local `kernel/build/` stage generates a signing key during kernel compilation.
2. That same key signs all NVIDIA `.ko` modules produced by `nvidia-modules-p2p-production-pkg`.
3. The installer image must boot a kernel from the **same build** (not just the same version ... the same actual compilation run).
4. If you rebuild modules but reuse an old installer, or vice versa, the key fingerprints won't match and modules silently fail to load.

### `CONFIG_MODULE_SIG_KEY` autogen trap

Copying a PEM into the kernel tree is not enough if the kernel config still points at the default autogenerated signing-key path. Kbuild treats the default path as eligible for regeneration and will still run the key-generation step.

The safe pattern is:

1. Generate one explicit shared signing key for the build run.
2. Point `CONFIG_MODULE_SIG_KEY` at a non-default filename.
3. Reuse that exact key across the kernel and NVIDIA module builds.

This repo's same-key workflow exists to enforce that rule consistently.

### Kernel payload mismatch is separate from key mismatch

Even if the signing-key flow is correct, the final installer can still be wrong if it embeds a stale kernel payload. Treat these as separate failure modes:

- **Signing-key mismatch**: modules are signed by a different key than the booted kernel trusts.
- **Kernel payload mismatch**: the installer contains an older kernel artifact than the one used to compile the modules.

Before upgrade, verify that the rebuilt kernel payload is propagated consistently through the kernel image, installer-base image, and final installer image.

**How to verify on a running node:**

```bash
# Check the booted kernel's signing key
talosctl read /proc/keys --nodes <ip> | grep -i asymmetric

# Check a loaded module's signature
talosctl read /sys/module/nvidia/holders --nodes <ip>
```

If the signing key in the module doesn't match the kernel's trusted keyring, you'll see the module present in `/usr/lib/modules/` but absent from `lsmod` output.

### The same-key rebuild chain

To guarantee lineage, the build must produce these artifacts from a single compilation:

1. `kernel` image (vmlinuz + modules with embedded signing key)
2. `installer-base` image (wraps the kernel into installer format)
3. `nvidia-modules-p2p-production-pkg` (compiled .ko files signed by that kernel's key)
4. `nvidia-modules-p2p-production` (sysext wrapper around the pkg)
5. Final installer (combines installer-base + all extensions)

Breaking this chain at any point produces a deployment that *looks* healthy but silently lacks GPU support.

## Extension Composition and Overlay Order

Talos system extensions are squashfs images that get overlaid onto the root filesystem. When multiple extensions provide the same file path, the **last extension in the list wins**.

The critical case: `ld.so.cache`.

- The `glibc` extension ships `/usr/local/glibc/etc/ld.so.cache` (1,931 bytes, no NVIDIA entries)
- The `nvidia-container-toolkit-production` extension ships the same path (6,699 bytes, 48 NVIDIA library entries)

If glibc overlays *after* toolkit, the NVIDIA entries vanish. CDI generation fails. The GPU operator crashes.

**The build script enforces correct ordering.** In `hack/p2p/build-installer.sh`, the toolkit is always added last to the `--system-extension-image` argument list. The script includes a post-build check verifying that the toolkit's index in `extensions.yaml` is higher than glibc's index.

### Verified extension order (working deployment)

| Index | Extension |
|---|---|
| 0 | nvidia-modules-p2p-production |
| 1 | glibc |
| 2 | binfmt-misc |
| 3 | btrfs |
| 4 | fuse3 |
| 5 | qemu-guest-agent |
| 6 | util-linux-tools |
| 7 | nvidia-container-toolkit-production |
| 8 | modules.dep |

Toolkit at 7, glibc at 1. Toolkit still wins because it overlays after glibc.

## ld.so.cache and Library Discovery

The NVIDIA container toolkit uses a multi-step library discovery chain:

1. If `--library-search-path` is provided, search those paths
2. Otherwise, search hardcoded paths: `/`, `/usr/lib64`, `/usr/lib/x86_64-linux-gnu`, `/lib64`, ...
3. Fall back to reading `/etc/ld.so.cache`

**Talos puts NVIDIA libraries at `/usr/local/glibc/usr/lib/`.** This is not in the hardcoded search paths and not in the default `ld.so.cache`.

The toolkit extension solves this by running `ldconfig -r /rootfs` at build time, which populates the cache with entries pointing to `/usr/local/glibc/usr/lib/`. The CDI generation service then uses `--library-search-path /usr/local/glibc/usr/lib` as a belt-and-suspenders measure.

### CDI generation service config

On Talos v1.13.0-beta.1, the CDI spec generation runs as a Talos service:
```
nvidia-ctk cdi generate \
  --library-search-path /usr/local/glibc/usr/lib \
  --output /run/cdi/nvidia.yaml
```

The output goes to `/run/cdi/` (tmpfs), so it's regenerated every boot.

## Official vs. Custom Boundaries

Only one thing is custom: the kernel modules.

| Component | Source | Custom? |
|---|---|---|
| Kernel modules (`nvidia.ko`, `nvidia_uvm.ko`, ...) | Local build with P2P patch | Yes |
| Container toolkit (`nvidia-ctk`, `nvidia-container-runtime`, ...) | Official `siderolabs/extensions` production track | No |
| Userspace libraries (`libcuda.so`, `libnvidia-ml.so`, ...) | Bundled inside official toolkit extension | No |
| Fabricmanager | Official `siderolabs/extensions` production track | No |
| Kernel | Local build (for signing key lineage only) | Yes, but functionally identical to stock |
| All other extensions (glibc, fuse3, btrfs, ...) | Official `siderolabs/extensions` | No |

Keep the custom surface area minimal. Rebuilding the kernel is a signing-key necessity, not a feature choice.

## NVIDIA Redist vs. GitHub Source Layout

Two common confusion points:

**The NVIDIA redist archive** (`nvidia_driver-linux-x86_64-595.58.03-archive.tar.xz`) is what the upstream build recipe uses. It extracts to a flat directory containing:
- `kernel-open/` ... open-source kernel module source
- `kernel/` ... proprietary kernel module source (binary blobs)
- Various userspace libraries and binaries

**The aikitoria donor patch** was created against the GitHub `open-gpu-kernel-modules` repo, where `kernel-open/` content lives at the repo root.

So the diff has two kinds of paths:
- `kernel-open/...` ... these apply directly against the redist extraction root
- `src/...` ... these need remapping to `src/kernel-module-source/src/...` in the redist layout

The P2P build recipe (`nvidia-gpu/nvidia-modules-p2p/kernel/production/pkg.yaml`) handles this path remapping during the patch-apply step.

## bldr Target Resolution

The `bldr` build tool discovers packages by walking the repo tree from `Pkgfile` at root:

- Target names come from the `name:` field in each `pkg.yaml`, **not** from directory paths
- Example: `nvidia-gpu/nvidia-modules-p2p/pkg/production/pkg.yaml` has `name: nvidia-modules-p2p-production`
- `.kres.yaml` lists target names; `kres` generates the Makefile TARGETS list
- Directory structure is organizational only

**Files that are path-sensitive** (and break if directories move):
- `hack/catalog.template` ... hardcodes all 12 category directory names as `BaseDir` strings
- `go.work` ... hardcodes 4 relative Go module paths
- `hack/p2p/check-userspace-track.sh` ... constructs `nvidia-gpu/vars.yaml` paths programmatically
- `docs/release-matrix.yaml` ... contains self-referential verification command paths

## Node Verification Flow

After deploying a new installer to a node, verify in this order:

1. **Node health:** `talosctl health --nodes <ip>` ... confirms the node booted and joined the cluster
2. **Extension list:** `talosctl get extensions --nodes <ip>` ... confirms both `nvidia-modules-p2p-production` and `nvidia-container-toolkit-production` are present
3. **Module loading:** `talosctl read /proc/modules --nodes <ip> | grep nvidia` ... confirms nvidia.ko, nvidia_uvm.ko, nvidia_drm.ko, nvidia_modeset.ko are loaded
4. **Driver params:** `talosctl read /proc/driver/nvidia/params --nodes <ip>` ... confirms the live `RegistryDwords` value includes the full P2P override string
5. **Kernel trust errors:** check `dmesg` for key-rejection messages such as `Loading of module with unavailable key is rejected` or `key was rejected by service`
6. **Driver version:** `talosctl read /proc/driver/nvidia/version --nodes <ip>` ... confirms the expected NVIDIA driver is actually loaded
7. **Library discovery:** `talosctl list /usr/local/glibc/usr/lib --nodes <ip> | grep libcuda` ... confirms userspace libraries are present
8. **CDI spec:** `talosctl read /run/cdi/nvidia.yaml --nodes <ip>` ... confirms CDI generation succeeded
9. **GPU operator:** `kubectl get pods -n gpu-operator` ... all pods should be Running/Completed
10. **Allocatable GPUs:** `kubectl get nodes -o json | jq '.items[].status.allocatable["nvidia.com/gpu"]'` ... should show the expected GPU count
11. **Runtime P2P check:** run `nvidia-smi topo -p2p r`, confirm CUDA peer access works, and verify `p2pBandwidthLatencyTest` succeeds

If step 3 fails (modules present on disk but not loaded), suspect a signing-key mismatch. If step 4 fails, suspect missing runtime registry overrides. If step 5 fails, suspect a signing or kernel-payload issue. If step 8 fails, suspect an `ld.so.cache` overlay-ordering problem.

## Ruled-Out Hypotheses

These explanations were investigated and are not the root cause of the current Talos P2P issue:

- the donor P2P patch being inherently broken on this GPU generation
- the VM machine type, OVMF, or guest IOMMU flags by themselves
- GPU operator bring-up alone
- allocatable GPU reporting alone
- BAR1 sizing alone

The durable conclusion is that the Talos-specific failure mode lived in the runtime bring-up path: signing lineage, installer/kernel consistency, extension overlay ordering, and the RM registry overrides all had to be correct at the same time.

## Known Gotchas

- **`PKGS` version format.** The Makefile's `PKGS` variable uses `git describe` format from the pkgs repo (e.g., `v1.13.0-beta.0-5-g142b074`), not the Talos release version. Don't confuse `v1.13.0-beta.1` (Talos) with `v1.13.0-beta.0-5-g142b074` (pkgs).

- **`make target-*` vs `make *`.** The custom P2P pkg doesn't get a top-level make target. Use `make target-nvidia-modules-p2p-production-pkg` (the generic target runner) instead of `make nvidia-modules-p2p-production-pkg` (which doesn't exist).

- **Installer-base requirements.** The stock `ghcr.io/siderolabs/installer-base:v1.13.0-beta.1` may not contain `/usr/install/<arch>/vmlinuz` after `docker load`. The custom `build-samekey-installer-base.py` script produces an installer-base that does.

- **BuildKit cache can preserve stale kernel or module outputs.** A superficially successful rebuild can still contain an older `nvidia.ko` or kernel payload if the relevant copy/build layers were reused. Prefer stage-scoped cache busting when validating rebuilt artifacts.

- **Pod scheduling can lag behind node readiness.** A node can return `Ready` before GPU validation pods are rescheduled. Treat scheduler lag as a separate timing issue, not immediate proof that the runtime fix failed.

- **Non-fatal build warnings.** You'll see `objtool: naked return found in MITIGATION_RETHUNK build` and `depmod: WARNING: nvidia-drm.ko needs unknown symbol drm_fbdev_ttm_driver_fbdev_probe` during compilation. These are benign.

- **`/var/run` is tmpfs.** CDI specs at `/run/cdi/nvidia.yaml` vanish on reboot. The Talos CDI generation service re-creates them at boot. Don't rely on persistent CDI specs.

- **GPU operator hook path.** On Talos v1.13+, the GPU operator device plugin expects `nvidia-ctk` at `/usr/bin/nvidia-ctk`, but Talos has it at `/usr/local/bin/nvidia-cdi-hook`. This is tracked in [siderolabs/talos#13021](https://github.com/siderolabs/talos/issues/13021).
